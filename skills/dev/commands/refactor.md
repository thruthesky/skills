# /dev:refactor — 리팩토링 분석 및 계획 수립

## ★ 최우선: .dev/DEV.md 가이드라인 확인

**이 명령을 실행하기 전에 반드시 `.dev/DEV.md` 파일이 존재하는지 확인한다.**

- `.dev/DEV.md`가 **존재하면**: 먼저 이 파일을 읽고, 그 안에 기록된 가이드라인에 따라 동작한다. DEV.md의 지시사항은 이 스킬 문서의 기본 동작보다 **우선**한다.
- `.dev/DEV.md`가 **존재하지 않으면**: 이 스킬 문서의 기본 동작 그대로 진행한다.

---

## 0. 필수 입력: ARGUMENTS

**이 명령을 사용하려면 반드시 ARGUMENTS를 제공해야 한다.**

ARGUMENTS가 없으면 사용자에게 다음을 요청하라:

```
/dev:refactor를 사용하려면 다음 정보를 프롬프트에 포함해 주세요:

필수:
  - 분석 대상: 어떤 코드/모듈/디렉터리를 분석할 것인가
    (전체 프로젝트, 특정 디렉터리, 특정 파일 등)

선택:
  - 저장 이름: .dev/<이름>/ 폴더에 저장할 이름
    (생략하면 분석 대상에서 자동 추론)
  - 초점 영역: 특정 분석에 집중 (보안 / 성능 / 중복 / 구조 등)
  - 심각도 기준: 어느 수준부터 보고할 것인가 (critical만 / 전체)
```

ARGUMENTS 예시:
```
예시 1: "/dev:refactor src/ — 전체 소스 코드 리팩토링 분석"
예시 2: "/dev:refactor app/Services/ 서비스계층 — 서비스 계층 중복/구조 분석"
예시 3: "/dev:refactor lib/widgets/ — 보안과 성능 위주로 분석해줘"
예시 4: "/dev:refactor 전체 — 프로젝트 전체 코드 품질 점검"
```

---

## 1. 이 명령의 목적

`/dev:refactor`는 기존 코드를 **CoT(Chain of Thought)** 와 **ToT(Tree of Thoughts)** 방식으로 심층 분석하여, `.dev/<폴더>/refactor-plan.md`에 **리팩토링 계획만** 작성한다.

**이 명령은 코드를 수정하지 않는다.** 계획만 만든다.
실제 리팩토링 실행은 `/dev:execute`로 한다.

핵심 원칙:

| 원칙 | 출처 | 적용 |
|---|---|---|
| **CoT 순차 분석** | AI Lang | 파일 → 함수 → 로직 순으로 순차적으로 추적하며 문제를 찾는다 |
| **ToT 다관점 분석** | AI Lang | 중복, 보안, 성능, 구조, 가독성 등 여러 관점에서 동시에 평가한다 |
| **완전한 명세** | SEDAI | 발견한 문제마다 현재 코드와 개선 방안을 구체적으로 명시한다 |
| **실행 가능한 계획** | Harness | Phase 단위로 나누어 `/dev:execute`로 바로 실행할 수 있는 수준으로 작성한다 |

## 2. 분석 프로세스

### 2.1 Phase 1 — CoT 순차 분석 (Chain of Thought)

코드를 순서대로 따라가며 분석한다:

```
1. 진입점 파일을 읽는다
2. 진입점에서 호출하는 모듈/클래스를 추적한다
3. 각 모듈 내부의 함수/메서드를 순서대로 분석한다
4. 데이터 흐름을 따라가며 변환/검증 과정을 확인한다
5. 외부 의존성과의 경계를 확인한다
```

각 단계에서 발견하는 것:
- 데이터가 검증 없이 사용되는 곳
- 에러 처리가 누락된 곳
- 로직이 불필요하게 복잡한 곳
- 하드코딩된 값
- 사용되지 않는 코드 (dead code)

### 2.2 Phase 2 — ToT 다관점 분석 (Tree of Thoughts)

동일한 코드를 **6가지 관점**에서 독립적으로 평가한다:

#### 관점 1: 중복 코드 (Duplication)
- 동일하거나 유사한 로직이 여러 곳에 존재하는가
- 복사-붙여넣기된 코드 블록이 있는가
- 공통 유틸리티로 추출할 수 있는 패턴이 있는가

#### 관점 2: 보안 취약점 (Security)
- SQL Injection 가능성
- XSS (Cross-Site Scripting) 가능성
- CSRF 보호 누락
- 인증/인가 검증 누락
- 민감 정보 노출 (API 키, 비밀번호, 토큰)
- 입력 검증 부재
- 파일 업로드 취약점
- 안전하지 않은 직렬화/역직렬화

#### 관점 3: 성능 문제 (Performance)
- N+1 쿼리 문제
- 불필요한 DB 조회
- 캐싱 미적용
- 메모리 누수 가능성
- 비효율적인 알고리즘/자료구조
- 불필요한 동기 처리 (비동기로 전환 가능)

#### 관점 4: 구조적 결함 (Architecture)
- 단일 책임 원칙(SRP) 위반
- 계층 간 의존성 방향 위반
- God Object / God Function
- 순환 의존성
- 추상화 수준 불일치
- 테스트 불가능한 구조

#### 관점 5: 가독성/유지보수성 (Readability)
- 과도하게 긴 함수 (50줄 이상)
- 과도하게 깊은 중첩 (3단계 이상)
- 의미 없는 변수/함수 이름
- 매직 넘버/문자열
- 주석 없는 복잡한 로직
- 일관되지 않은 코딩 스타일

#### 관점 6: 불필요한 코드 (Dead Code)
- 호출되지 않는 함수/메서드
- 도달할 수 없는 코드 블록
- 사용되지 않는 import/변수
- 주석 처리된 코드
- 더 이상 필요 없는 호환성 코드

### 2.3 Phase 3 — 종합 및 우선순위 결정

6가지 관점의 분석 결과를 종합하여 우선순위를 매긴다:

| 심각도 | 기준 | 예시 |
|---|---|---|
| **Critical** | 보안 취약점, 데이터 손실 위험 | SQL Injection, 인증 우회 |
| **High** | 성능 심각 저하, 구조적 결함 | N+1 쿼리, God Object |
| **Medium** | 중복 코드, 유지보수 어려움 | 복사-붙여넣기 로직, 긴 함수 |
| **Low** | 가독성, 코딩 스타일 | 변수 이름, 매직 넘버 |
| **Info** | 개선하면 좋지만 긴급하지 않음 | Dead code, 주석 처리된 코드 |

## 3. 리팩토링 계획 형식

`.dev/<폴더>/refactor-plan.md`에 다음 형식으로 작성한다:

```markdown
# <대상> 리팩토링 계획

상태: Draft | Reviewed | Ready
작성일: YYYY-MM-DD
분석 대상: <파일/디렉터리 경로>
분석 방법: CoT 순차 분석 + ToT 6관점 분석

---

|--------|------|----------|
| Critical | N | ... |
| High | N | ... |
| Medium | N | ... |
| Low | N | ... |
| Info | N | ... |

총 발견 건수: N건

- **현재 코드**:
  ```<언어>

  ```
- **영향 범위**: 이 변경이 영향을 주는 다른 파일/모듈

- 검증 방법

### Phase 2: High 이슈 해결
- 작업 내용

- 검증 방법

### Phase 3: Medium 이슈 해결
- 작업 내용
- 대상 파일

## Constraints
### must
- 기존 기능이 깨지지 않아야 한다
- 기존 API 인터페이스를 유지해야 한다

### must_not
- 리팩토링 과정에서 새 기능을 추가하지 않는다
- 리팩토링 범위를 벗어난 코드를 수정하지 않는다

---

- [ ] High 이슈 전부 해결

- [ ] 새로 추가된 테스트 통과

## 4. 분석 규칙

### 4.1 코드를 반드시 직접 읽어라

- 추측으로 문제를 나열하지 않는다
- 실제 파일을 읽고, 실제 코드에서 문제를 찾는다
- 파일 경로와 라인 번호를 구체적으로 명시한다

### 4.2 현재 코드와 개선 코드를 함께 보여줘라 (SEDAI 원칙)

각 발견 사항에 반드시 포함:
- **현재 코드**: 문제가 있는 실제 코드 스니펫
- **개선 방안**: 구체적인 개선 코드 또는 의사코드

"이 부분을 개선하세요" 같은 모호한 지시 금지. 정확히 어떻게 바꿔야 하는지 명시.

### 4.3 영향 범위를 명시하라

코드 변경 시 영향받는 다른 파일/모듈을 반드시 기록한다:
- 이 함수를 호출하는 곳
- 이 클래스를 상속/구현하는 곳
- 이 API를 사용하는 클라이언트

### 4.4 거짓 양성(false positive)을 피하라

- 의도적인 설계 결정을 문제로 지적하지 않는다
- 프레임워크 관례를 위반 사항으로 보고하지 않는다
- 확신이 없으면 "확인 필요"로 표시한다

## 5. 실전 작업 흐름

```
1. ARGUMENTS에서 분석 대상과 저장 이름을 파악한다
2. .dev/DEV.md가 있으면 먼저 읽는다
3. 분석 대상 코드를 읽는다
4. CoT 순차 분석을 수행한다 (진입점 → 호출 추적 → 로직 분석)
5. ToT 6관점 분석을 수행한다 (중복/보안/성능/구조/가독성/Dead Code)
6. 발견 사항을 심각도별로 분류한다
7. 리팩토링 실행 계획을 Phase별로 작성한다
8. .dev/<이름>/refactor-plan.md에 저장한다
9. 사용자에게 분석 요약을 보고한다
```

## 6. 완료 기준

- [ ] 분석 대상 코드를 실제로 읽었는가
- [ ] CoT 순차 분석을 수행했는가
- [ ] ToT 6관점 분석을 모두 수행했는가
- [ ] 각 발견 사항에 현재 코드와 개선 방안이 있는가
- [ ] 파일 경로와 라인 번호가 명시되어 있는가
- [ ] 심각도별로 분류되어 있는가
- [ ] Phase별 실행 계획이 있는가
- [ ] `/dev:execute`로 바로 실행할 수 있는 수준인가

**코드를 직접 읽지 않고 추측으로 작성한 분석은 분석이 아니다.**
