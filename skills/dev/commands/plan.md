# /dev:plan — 구현 명세서 작성

## ★ 최우선: .dev/DEV.md 가이드라인 확인

**이 명령을 실행하기 전에 반드시 `.dev/DEV.md` 파일이 존재하는지 확인한다.**

- `.dev/DEV.md`가 **존재하면**: 먼저 이 파일을 읽고, 그 안에 기록된 가이드라인에 따라 동작한다. DEV.md의 지시사항은 이 스킬 문서의 기본 동작보다 **우선**한다.
- `.dev/DEV.md`가 **존재하지 않으면**: 이 스킬 문서의 기본 동작 그대로 진행한다.

DEV.md는 프로젝트별 커스텀 가이드라인이다. 예를 들어 plan 시 특정 명세 형식, 추가 섹션, 채점 기준 변경 등의 지시가 있을 수 있다.

---

## 0. 필수 입력: ARGUMENTS

**이 명령을 사용하려면 반드시 ARGUMENTS를 제공해야 한다.**

ARGUMENTS가 없으면 사용자에게 다음을 요청하라:

```
/dev:plan을 사용하려면 다음 정보를 프롬프트에 포함해 주세요:

필수:
  - 구현할 기능 또는 프로젝트에 대한 설명

선택:
  - 저장 이름: .dev/<이름>/ 폴더에 저장할 이름
    (생략하면 기능 설명에서 자동 추론하여 이름을 결정)
  - 기술 스택 (언어, 프레임워크, DB 등)
  - 참조할 기존 .dev/ 문서 (이전에 /dev:copy로 만든 것)
  - 구현 제약사항 (반드시 지켜야 할 규칙)
  - 기대 결과물 (어떤 상태가 완료인지)
```

### 저장 이름 결정 규칙

사용자가 이름을 지정하면 그 이름을 사용한다.
사용자가 이름을 생략하면 기능 설명에서 자동 추론한다:
- 핵심 키워드를 kebab-case로 변환 (예: "게시판 기능" → `board`, "JWT 인증" → `jwt-auth`)
- 한글도 허용하되 짧게 (예: "채팅", "게시판")

ARGUMENTS 예시:
```
예시 1: "/dev:plan 게시판 — Laravel로 게시글 CRUD + 댓글 + 좋아요 기능"
예시 2: "/dev:plan — Flutter + Firebase Firestore 실시간 채팅 → 이름 생략 → .dev/chat/ 에 저장"
예시 3: "/dev:plan 인증리팩토링 — 기존 세션 기반 인증을 JWT로 전환. .dev/현재인증/ 참조"
```

## 1. 이 명령의 목적

`/dev:plan`은 사용자의 요청을 **완전한 구현 명세서**로 작성하여 `.dev/<폴더>/plan.md`에 저장한다.

이 명세서는 `/dev:execute`가 읽고 그대로 실행할 수 있는 수준이어야 한다.

핵심 원칙 (3가지 개념 융합):

| 원칙 | 출처 | 적용 |
|---|---|---|
| **구조화된 명세** | AI Lang | intent, context, constraints, output 등 YAML 구조로 명세의 뼈대를 잡음 |
| **완전성 + 채점** | SEDAI | 명세서 완전성을 0~100점으로 자체 채점. 95점 미만이면 부족한 부분을 명시하고 사용자에게 보완 요청 |
| **실행 계획 구조** | Harness | Work Phases, Validation Plan, Definition of Done으로 실행 가능한 수준까지 구체화 |

핵심 특징:

| 항목 | 설명 |
|------|------|
| 분석 방식 | **에이전트 팀/서브 에이전트 병렬 분석** |
| 분석 깊이 | **3개 관점 독립 탐색 + 비평 에이전트 통합** |
| 비주얼 산출물 | **ASCII 다이어그램 필수 (아키텍처 + 흐름도)** |
| 리스크 분석 | **Tree-of-Thoughts 방식 심층 리스크 분석** |
| 최종 산출물 | `.dev/<이름>/plan.md` |
| 명세 점수 목표 | **95점 이상 필수 (미달 시 보완 후 재작성)** |

## 3. 명세서 형식

`.dev/<폴더>/plan.md`에 다음 형식으로 작성한다:

```markdown
# <기능/프로젝트 이름> 구현 명세서

상태: Draft | Reviewed | Ready | Active | Completed
작성일: YYYY-MM-DD
명세 점수: __/100

---

## Intent
- 구현 목적을 한 문장으로 (AI Lang의 intent)
- intent 유형: CREATE | UPDATE | TRANSFORM | MIGRATE | DEBUG | REFACTOR

## Context
- domain: <도메인>
- language: <언어>
- framework: <프레임워크>
- database: <DB>
- 관련 .dev/ 참조: [<이름>](./<경로>)

## Current State (현재 상태)
- 지금 구현되어 있는 것
- 현재의 문제점이나 부족한 점
- (없으면 "신규 구현" 명시)

## Expected Result (기대 결과)
- 구현 완료 후 달성해야 할 상태
- 사용자가 할 수 있게 되는 것

## Scope (범위)
### 포함
- 이 명세서가 다루는 것

### 제외 (Non-Goals)
- 이 명세서가 다루지 않는 것

## Users and Roles
- 관련 사용자 유형과 역할

## Visual Diagrams (비주얼 다이어그램)
### 아키텍처 다이어그램
- ASCII 형식으로 계층 구조, 파일 구조, 의존성 관계 시각화

### 흐름도 (Flow Chart)
- ASCII 형식으로 사용자 흐름 또는 데이터 흐름 step-by-step 시각화

### 추가 다이어그램 (필요 시)
- 시퀀스 다이어그램: 클라이언트↔서버↔DB 간 요청/응답 순서
- 데이터 모델 (ERD): Entity 간 관계, 필드, FK
- 상태 다이어그램: 상태 전이 (예: 주문 상태, 신고 처리 상태)
- 컴포넌트 다이어그램: 프론트엔드 컴포넌트 구조

## Key Flows (핵심 흐름)
### 흐름 1: <이름>
1. 단계별 사용자/시스템 행동
2. ...

### 흐름 2: <이름>
1. ...

## Entities (데이터 모델)
- 엔티티 이름, 필드, 관계
- 테이블 스키마 또는 타입 정의

## Architecture (구현 아키텍처)
### 파일 구조
- 생성/수정할 파일 목록과 역할

### 계층 구조
- 컨트롤러 → 서비스 → 리포지토리 등 의존성 방향

### 핵심 구현 코드
- 주요 함수/클래스의 시그니처와 핵심 로직
- 의사코드 또는 실제 코드 스니펫

## Constraints (제약사항)
### must (반드시)
- 반드시 지켜야 하는 규칙

### should (권장)
- 가능하면 지켜야 하는 규칙

### must_not (금지)
- 절대 하지 말아야 하는 것

## Work Phases (작업 단계)
### Phase 1: <이름>
- 작업 내용
- 산출물
- 검증 방법

### Phase 2: <이름>
- ...

## Validation Plan (검증 계획)
- 각 단계별 무엇을 어떻게 검증하는가
- 테스트 케이스 목록

## Test Plan (테스트 계획) — 필수
### 프로젝트 유형
- Web | Flutter | Backend API | CLI | Mobile Native | 기타

### 사용 가능한 테스트 도구 / MCP 확인 결과
명세서 작성 시점에 다음을 반드시 조사하여 기록한다:
- Playwright MCP: 사용 가능 / 사용 불가
- Chrome DevTools MCP: 사용 가능 / 사용 불가
- 기타 사용 가능한 MCP (Slack, GitHub, DB 등): ...
- 로컬 테스트 러너: <pytest / phpunit / vitest / flutter test 등>

### 자동화 테스트
```

## 4. 명세서 작성 프로세스

### 4.1 Phase 1: 프로젝트 컨텍스트 수집 (필수)

#### 필수 참조 문서 읽기

다음 문서들을 **반드시** 읽고 프로젝트 현재 상태를 완전히 파악한다:

| 순서 | 대상 | 파악할 내용 |
|------|------|------------|
| 1 | `CLAUDE.md` | 기술 스택, 아키텍처 원칙, 코딩 규칙, 필수 워크플로우, 타입 검사 규칙 |
| 2 | **현재 프로젝트의 스킬** (`SKILL.md` + `references/*.md`) | 프로젝트 전체 구조, 참조 문서 인덱스, 핵심 패턴. CLAUDE.md에 프로젝트 스킬 경로가 명시되어 있으면 해당 스킬을 읽고, 없으면 `.claude/skills/` 하위를 탐색하여 관련 스킬을 찾는다 |
| 3 | `.dev/` 디렉토리 | 기존 참조/계획 문서 중 ARGUMENTS와 관련된 것 |
| 4 | 관련 소스코드 | Controller, Service, Repository, Entity, 페이지 등 실제 코드 |
| 5 | 기존 `.dev/` 참조 | 이전에 `/dev:copy`로 만든 참조 문서가 있으면 반드시 읽는다 |
| 6 | 기술 스택 확인 | 패키지 매니저 파일, 설정 파일 등을 읽어 실제 스택을 파악한다 |

**프로젝트 스킬 탐색 규칙:**
1. CLAUDE.md를 먼저 읽어 프로젝트 설명과 스킬 경로가 있는지 확인한다
2. 프로젝트 스킬이 있으면 `SKILL.md`와 관련 `references/*.md`를 읽어 프로젝트 전체 구조를 파악한다
3. 프로젝트 스킬이 없으면 CLAUDE.md의 정보와 소스코드 직접 탐색으로 컨텍스트를 수집한다

#### 프로젝트 현재 상태 종합

수집한 정보를 다음 체크리스트로 정리한다:

```
[ ] 기술 스택 확인 (언어, 프레임워크, DB, 프론트엔드 등 — CLAUDE.md에서 파악)
[ ] 아키텍처 레이어 파악 (MVC, Clean Architecture, 모듈 구조 등)
[ ] 프로젝트 고유 패턴 이해 (인증, 멀티테넌트, 라우팅 등 — 프로젝트 스킬에서 파악)
[ ] ARGUMENTS 관련 기존 코드/문서 파악 완료
[ ] 영향받는 파일 목록 식별
```

### 4.2 Phase 2: 에이전트 팀 Deep Analysis (CoT + ToT 병렬 탐색)

ARGUMENTS를 **3개의 독립적인 관점**에서 **에이전트 팀을 활용하여 병렬로** 탐색하고, **1개의 비평 에이전트**가 결과를 통합·평가한다.

#### 에이전트 실행 전략

```
┌─────────────────────────────────────────────────────────────────┐
│  에이전트 팀 실행 전략 (우선순위 순서)                             │
│                                                                 │
│  1순위: 에이전트 팀(Agent Team)이 사용 가능하면                   │
│         → 에이전트 팀으로 3개 에이전트를 동시 실행                │
│                                                                 │
│  2순위: 에이전트 팀이 불가능하면                                  │
│         → 서브 에이전트(Agent 도구)를 3개 동시 호출하여           │
│           병렬로 분석 수행                                       │
│                                                                 │
│  ※ 어떤 경우든 3개 에이전트는 반드시 병렬로 실행해야 한다.       │
│    순차 실행은 금지된다.                                         │
└─────────────────────────────────────────────────────────────────┘
```

#### 병렬 탐색 에이전트 (3개 동시 실행)

```
┌─────────────────────────────────────────────────────────────────────┐
│                        ARGUMENTS 분석                               │
├─────────────────┬─────────────────────┬─────────────────────────────┤
│                 │                     │                             │
│  Agent A        │  Agent B            │  Agent C                   │
│  [아키텍처]     │  [데이터/흐름]      │  [리스크/엣지케이스]       │
│                 │                     │                             │
│  ● 아키텍처에서 │  ● 데이터 모델      │  ● 보안 취약점             │
│    어디에 위치?  │    (Entity/필드)     │  ● 성능 병목               │
│  ● 기존 코드와  │  ● 사용자 흐름      │  ● 프로젝트 고유 제약 위반 │
│    의존성 관계   │    (step-by-step)   │  ● 타입 안전성 위험        │
│  ● 파일 생성/   │  ● API 엔드포인트   │  ● 동시성/경합 조건        │
│    수정 목록     │  ● DB 마이그레이션  │  ● 기존 기능 회귀          │
│  ● 라우팅 변경  │  ● 검색/캐시 영향   │  ● 잘못된 가정 목록        │
│                 │                     │                             │
└─────────────────┴─────────────────────┴─────────────────────────────┘
```

**각 에이전트는 Chain-of-Thought(CoT) 방식으로 단계별 추론한다.**

##### Agent A: 아키텍처 분석 (CoT)

```
사고 체인:
1. ARGUMENTS가 요구하는 기능은 무엇인가?
2. 현재 아키텍처 구조에서 어디에 위치하는가?
   → 각 레이어(Controller/Service/Repository/Entity 등)에 변경이 필요한가?
3. 기존 코드 중 재사용 가능한 것은?
4. 새로 생성해야 할 파일은?
5. 기존 파일 중 수정이 필요한 것은?
6. 라우팅에 변경이 필요한가?
7. 프론트엔드(페이지/컴포넌트/스타일) 변경은?
```

##### Agent B: 데이터/흐름 분석 (CoT)

```
사고 체인:
1. 어떤 데이터가 필요한가? (Entity, 테이블, 필드)
2. DB 마이그레이션이 필요한가?
3. 사용자 흐름은 어떻게 되는가? (step 1 → step 2 → ...)
4. API 엔드포인트가 필요한가? (메서드, URL, 파라미터, 응답)
5. 검색/캐시/인덱스에 영향이 있는가?
6. 알림(푸시, 이메일 등)이 필요한가?
7. 다국어(i18n) 문자열이 필요한가?
```

##### Agent C: 리스크/엣지케이스 분석 (Tree-of-Thoughts)

```
사고 트리:
├── 보안
│   ├── XSS 가능성은?
│   ├── SQL Injection 가능성은?
│   ├── CSRF 대응은?
│   └── 인증/권한 우회 가능성은?
├── 프로젝트 고유 제약
│   ├── CLAUDE.md에 명시된 규칙을 모두 준수하는가?
│   ├── 프로젝트 스킬에 정의된 패턴과 일치하는가?
│   └── 기존 아키텍처 원칙에 위배되지 않는가?
├── 타입 안전성
│   ├── nullable 반환값 목록은?
│   ├── union 타입 처리가 필요한 곳은?
│   └── 타입 에러 발생 가능 지점은?
├── 성능
│   ├── N+1 쿼리 가능성은?
│   ├── 대량 데이터 처리 시 문제는?
│   └── 인덱스가 필요한 쿼리는?
└── 회귀
    ├── 기존 테스트에 영향은?
    ├── 기존 API 호환성은?
    └── 기존 UI/UX에 변경은?
```

#### 비평 에이전트 (통합·평가)

3개 에이전트의 분석 결과를 받아 다음을 수행한다:

```
┌────────────────────────────────────────────────────────┐
│  비평 에이전트 — 통합 평가                              │
│                                                        │
│  1. 3개 분석 결과 간 모순 검출                          │
│     → 모순이 있으면 근거를 비교하여 올바른 쪽 채택      │
│                                                        │
│  2. 누락 항목 식별                                      │
│     → 3개 에이전트 모두 놓친 관점이 있는지 확인         │
│                                                        │
│  3. 우선순위 정리                                       │
│     → Critical / High / Medium / Low로 분류             │
│                                                        │
│  4. 실행 가능성 판단                                    │
│     → 현재 프로젝트 상태에서 구현 가능한지 평가         │
│     → 선행 작업(마이그레이션 등)이 필요한지 식별        │
│                                                        │
│  5. 통합 분석 보고서 생성                               │
│     → Phase 3의 입력 데이터로 사용                      │
└────────────────────────────────────────────────────────┘
```

### 4.3 Phase 3: 비주얼 다이어그램 생성 + 명세서 작성

#### 비주얼 다이어그램 생성 (필수)

Phase 2의 분석 결과를 바탕으로 **최소 2개 이상**의 다이어그램을 생성한다.

복잡한 구현 계획을 텍스트로만 설명하면 이해하기 어렵다. 다이어그램을 통해 사용자가 계획의 전체 구조와 흐름을 **한눈에** 파악할 수 있도록 한다.

##### 필수 다이어그램 유형

| 다이어그램 | 형식 | 용도 |
|-----------|------|------|
| **아키텍처 다이어그램** | ASCII | 계층 구조, 파일 구조, 의존성 관계 시각화 |
| **흐름도 (Flow Chart)** | ASCII | 사용자 흐름 또는 데이터 흐름 step-by-step 시각화 |

##### 선택 다이어그램 유형 (필요 시 추가)

| 다이어그램 | 형식 | 용도 |
|-----------|------|------|
| 시퀀스 다이어그램 | ASCII | 클라이언트↔서버↔DB 간 요청/응답 순서 |
| 데이터 모델 (ERD) | ASCII | Entity 간 관계, 필드, FK |
| 상태 다이어그램 | ASCII | 상태 전이 (예: 주문 상태, 신고 처리 상태) |
| 컴포넌트 다이어그램 | ASCII | 프론트엔드 컴포넌트 구조 |

##### ASCII 다이어그램 예시

**아키텍처 다이어그램 예시:**
```
                    ┌──────────────┐
                    │   Browser    │
                    └──────┬───────┘
                           │ HTTP Request
                           ▼
                    ┌──────────────┐
                    │   Router     │  (라우팅)
                    └──────┬───────┘
                           │
              ┌────────────┼────────────┐
              ▼            ▼            ▼
       ┌────────────┐ ┌────────┐ ┌──────────┐
       │ Controller │ │ Pages  │ │ Partials │
       │ (API)      │ │ (View) │ │ (Layout) │
       └─────┬──────┘ └────────┘ └──────────┘
             │
             ▼
       ┌────────────┐
       │  Service   │ ← 비즈니스 로직
       └─────┬──────┘
             │
             ▼
       ┌────────────┐
       │ Repository │ ← DB 접근
       └─────┬──────┘
             │
             ▼
       ┌────────────┐
       │  Entity    │ ← 데이터 모델
       └────────────┘
```

**흐름도 예시:**
```
  [사용자 액션]
       │
       ▼
  ┌─ 인증 확인 ─┐
  │              │
  ▼ (인증됨)    ▼ (미인증)
  권한 확인      401 반환
  │              
  ▼ (권한 있음)  
  입력값 검증    
  │              
  ├─ 유효 ──▶ Service 호출 ──▶ Repository ──▶ DB
  │                                           │
  │                                           ▼
  │                                      성공 응답
  │
  └─ 무효 ──▶ 422 반환
```

#### 명세서 초안 작성

Phase 2의 통합 분석 보고서 + 비주얼 다이어그램을 기반으로, 섹션 3의 형식에 따라 `.dev/<폴더>/plan.md`를 작성한다.

**핵심 규칙:**
- 모호한 표현 금지. "적절히 처리한다" 대신 정확히 어떻게 처리하는지 명시
- 핵심 함수/클래스의 시그니처와 로직을 코드 수준으로 기술
- 파일 경로를 구체적으로 명시
- Current State와 Expected Result의 차이가 명확해야 한다
- Phase 2 에이전트 팀 Deep Analysis 결과 반영
- 리스크/엣지케이스 분석 결과 Risks 섹션에 상세 기술
- 프로젝트 스킬 참조 문서와의 정합성 검증 결과 기술
- CLAUDE.md 규칙 준수 계획 명시

### 4.4 자체 채점 (SEDAI 원칙)

명세서 작성 후 반드시 **자체 채점**을 수행한다.

채점 기준:

| 항목 | 배점 | 기준 |
|---|---|---|
| Intent 명확성 | 10 | 무엇을 왜 만드는지 한 문장으로 설명 가능한가 |
| Context 완전성 | 10 | 기술 스택, 참조 문서, 의존성이 모두 명시되었는가 |
| Key Flows 구체성 | 10 | 사용자 흐름이 단계별로 구체적인가 |
| Entities 완전성 | 10 | 데이터 모델, 필드, 관계가 모두 정의되었는가 |
| Architecture 실행성 | 15 | 파일 구조, 계층, 핵심 코드가 포함되었는가 |
| Constraints 명확성 | 5 | must/should/must_not이 구체적인가 |
| Work Phases 실행성 | 10 | 각 단계의 산출물과 검증 방법이 있는가 |
| **Test Plan 검증성** | **15** | **사용 가능한 MCP/도구 확인, 시나리오 구체성, 통과 기준 명확** |
| Acceptance Criteria | 10 | 완료 기준이 체크리스트로 검증 가능한가 |
| 전체 일관성 | 5 | 문서 내 모순이나 누락이 없는가 |

**95점 이상: Ready** — `/dev:execute`로 실행 가능
**80~94점: Reviewed** — 부족한 부분을 명시하고 사용자에게 보완 요청
**80점 미만: Draft** — 핵심 정보가 부족. 추가 정보를 반드시 요청

### 4.5 부족한 부분 처리

채점 결과 95점 미만이면:

1. 부족한 항목을 명확히 나열한다
2. 사용자에게 보완할 정보를 구체적으로 요청한다
3. 추측으로 채우지 않는다 (SEDAI 원칙: "No interpretation, no assumption")
4. 사용자가 보완 정보를 제공하면 명세서를 업데이트하고 재채점한다

예시:
```
명세 완전성 점수: 82/100

부족한 부분:
- Entities (-8): 댓글 테이블의 필드 정의가 누락됨. parent_id로 대댓글을 지원할 것인지 확인 필요
- Architecture (-5): 이미지 업로드 시 저장소(로컬/S3/Firebase Storage) 미결정
- Constraints (-5): 게시글 본문의 최대 길이, 이미지 최대 개수 미정의

다음 정보를 알려주시면 95점 이상으로 보완할 수 있습니다:
1. 대댓글 지원 여부와 최대 깊이
2. 이미지 저장소 선택
3. 본문 최대 길이와 이미지 최대 개수
```

## 6. 보조 문서 생성

명세서가 복잡할 경우 보조 문서를 추가로 생성한다:

```text
.dev/<폴더>/
├── plan.md                 ← 메인 구현 명세서 (필수)
├── entities.md             ← 상세 데이터 모델 (plan.md에서 링크)
├── flows.md                ← 상세 사용자 흐름 (plan.md에서 링크)
├── api-spec.md             ← API 엔드포인트 명세 (plan.md에서 링크)
└── ui-spec.md              ← 화면 명세 (plan.md에서 링크)
```

보조 문서를 만드는 기준:
- plan.md가 1700줄을 초과할 때
- 특정 영역(데이터 모델, API 등)이 독립적으로 참조될 필요가 있을 때
- 메인 문서에서 Progressive Disclosure로 연결하는 것이 더 읽기 좋을 때

## 7. 실전 작업 흐름

```
Step 1  ARGUMENTS에서 구현 대상과 저장 이름을 파악한다

Step 2  Phase 1 실행 — 프로젝트 컨텍스트 수집
        ├─▶ CLAUDE.md 읽기 (기술 스택, 규칙 파악)
        ├─▶ 현재 프로젝트의 스킬 SKILL.md + 관련 references/*.md 읽기
        ├─▶ .dev/ 기존 문서 확인
        ├─▶ 관련 소스코드 읽기
        └─▶ 기술 스택 확인 (패키지 매니저 파일, 설정 파일)

Step 3  Phase 2 실행 — 에이전트 팀 Deep Analysis
        ├─▶ Agent A (아키텍처 CoT) ─┐
        ├─▶ Agent B (데이터/흐름 CoT) ├─▶ 비평 에이전트 통합
        └─▶ Agent C (리스크 ToT)  ───┘
        ※ 에이전트 팀이 불가능하면 서브 에이전트 3개를 병렬 호출

Step 4  Phase 3 실행 — 비주얼 다이어그램 생성
        ├─▶ 아키텍처 다이어그램 (ASCII)
        ├─▶ 흐름도 (ASCII)
        └─▶ 추가 다이어그램 (필요 시)

Step 5  프로젝트 유형(웹/Flutter/백엔드/CLI 등)을 결정한다

Step 6  사용 가능한 MCP 도구를 확인한다 (Playwright MCP, Chrome DevTools MCP 등)

Step 7  .dev/<이름>/ 디렉터리를 생성한다

Step 8  plan.md 명세서를 작성한다 (비주얼 다이어그램 + Test Plan 섹션 필수 포함)

Step 9  자체 채점한다 (0~100점)

Step 10 95점 이상이면 "Ready"로 표시한다

Step 11 95점 미만이면 부족한 부분을 명시하고 사용자에게 보완을 요청한다

Step 12 보완 후 재채점한다

Step 13 ★ 완료 보고 시 사용자에게 반드시 `/dev:review`로 적대적 리뷰를 수행할 것을 요청한다
```

## 8. 완료 기준

- [ ] Phase 1 — CLAUDE.md, 프로젝트 스킬 참조 문서, 관련 소스코드를 모두 읽었는가
- [ ] Phase 2 — 3개 에이전트 병렬 분석 + 비평 에이전트 통합이 완료되었는가
- [ ] Phase 2 — 에이전트 팀 또는 서브 에이전트를 사용하여 병렬로 실행했는가
- [ ] Phase 3 — ASCII 다이어그램 최소 2개(아키텍처 + 흐름도)가 생성되었는가
- [ ] `.dev/<이름>/plan.md`가 존재하는가
- [ ] Intent가 한 문장으로 명확한가
- [ ] Key Flows가 단계별로 구체적인가
- [ ] Entities가 필드 수준으로 정의되었는가
- [ ] Architecture에 파일 구조와 핵심 코드가 포함되었는가
- [ ] Constraints가 must/should/must_not으로 구분되었는가
- [ ] Work Phases에 각 단계의 산출물과 검증 방법이 있는가
- [ ] **Test Plan에 사용 가능한 MCP/테스트 도구와 검증 시나리오가 명시되었는가**

- [ ] **Flutter 프로젝트라면 단위/위젯/통합 테스트와 디바이스 검증 시나리오가 모두 포함되었는가**

## 9. 작성 완료 후 — 반드시 리뷰 요청

명세서 작성과 자체 채점이 끝나면, **반드시 사용자에게 `/dev:review` 명령으로 이 명세서를 적대적(Adversarial) 관점에서 리뷰할 것을 권장한다.**

자체 채점만으로는 다음과 같은 사각지대를 발견하지 못한다:
- 자기 확증 편향 (Confirmation bias)
- 미처 생각하지 못한 엣지 케이스
- 보안/성능/설계의 잠재적 결함
- 가정에 대한 검증 부재
- 누락된 비기능 요구사항

### 사용자에게 보내는 완료 보고 메시지 형식

명세서를 작성한 후 사용자에게 다음과 같이 보고한다:

```
✅ .dev/<폴더>/plan.md 명세서 작성 완료
- 명세 점수: __/100
- 상태: Ready | Reviewed | Draft

⚠️ 다음 단계: 반드시 적대적 리뷰를 수행하세요.

다음 명령으로 리뷰를 실행할 수 있습니다:

  /dev:review .dev/<폴더>/plan.md — 이 계획이 실행 가능한지 가차 없이 리뷰해줘

리뷰 결과는 .dev/<폴더>/review.md 에 저장되며,
적대적 관점에서 발견된 결함/문제/보강사항이 정리됩니다.
이를 바탕으로 plan.md를 보완한 후 /dev:execute로 구현을 시작하세요.
```

이 단계를 생략하지 말 것. 리뷰를 거치지 않은 명세서는 실행 단계에서 큰 비용을 초래할 수 있다.

## 10. 주의사항

1. **절대 코드를 작성하지 않는다** — 이 명령은 계획 수립 전용이다. 코드 구현은 `/dev:execute`로 별도 실행한다.
2. **추측으로 채우지 않는다** — 정보가 부족하면 사용자에게 구체적으로 질문한다 (SEDAI 원칙).
3. **CLAUDE.md 규칙을 반드시 준수한다** — CLAUDE.md에 명시된 타입 검사, 아키텍처 원칙, 코딩 규칙 등이 계획에 반영되어야 한다.
4. **프로젝트 스킬이 있으면 반드시 참조한다** — 프로젝트 스킬의 패턴, 로직, 구조를 계획에 일관되게 반영한다.
5. **다이어그램은 ASCII로 작성한다** — 터미널/마크다운 환경에서 바로 읽을 수 있어야 한다.
6. **에이전트 병렬 실행은 필수이다** — 에이전트 팀이 가능하면 에이전트 팀을, 불가능하면 서브 에이전트를 병렬로 사용하여 3개 관점을 동시에 분석한다.
7. **비주얼 다이어그램은 필수이다** — 사용자가 계획의 전체 구조와 흐름을 한눈에 파악할 수 있도록 아키텍처 다이어그램과 흐름도를 반드시 포함한다.
