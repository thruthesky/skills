# /dev:plan — 구현 명세서 작성

## 0. 필수 입력: ARGUMENTS

**이 명령을 사용하려면 반드시 ARGUMENTS를 제공해야 한다.**

ARGUMENTS가 없으면 사용자에게 다음을 요청하라:

```
/dev:plan을 사용하려면 다음 정보를 프롬프트에 포함해 주세요:

필수:
  - 구현할 기능 또는 프로젝트에 대한 설명

선택:
  - 저장 이름: .dev/<이름>/ 폴더에 저장할 이름
    (생략하면 기능 설명에서 자동 추론하여 이름을 결정)
  - 기술 스택 (언어, 프레임워크, DB 등)
  - 참조할 기존 .dev/ 문서 (이전에 /dev:copy로 만든 것)
  - 구현 제약사항 (반드시 지켜야 할 규칙)
  - 기대 결과물 (어떤 상태가 완료인지)
```

### 저장 이름 결정 규칙

사용자가 이름을 지정하면 그 이름을 사용한다.
사용자가 이름을 생략하면 기능 설명에서 자동 추론한다:
- 핵심 키워드를 kebab-case로 변환 (예: "게시판 기능" → `board`, "JWT 인증" → `jwt-auth`)
- 한글도 허용하되 짧게 (예: "채팅", "게시판")

ARGUMENTS 예시:
```
예시 1: "/dev:plan 게시판 — Laravel로 게시글 CRUD + 댓글 + 좋아요 기능"
예시 2: "/dev:plan — Flutter + Firebase Firestore 실시간 채팅 → 이름 생략 → .dev/chat/ 에 저장"
예시 3: "/dev:plan 인증리팩토링 — 기존 세션 기반 인증을 JWT로 전환. .dev/현재인증/ 참조"
```

---

## 1. 이 명령의 목적

`/dev:plan`은 사용자의 요청을 **완전한 구현 명세서**로 작성하여 `.dev/<폴더>/plan.md`에 저장한다.

이 명세서는 `/dev:execute`가 읽고 그대로 실행할 수 있는 수준이어야 한다.

핵심 원칙 (3가지 개념 융합):

| 원칙 | 출처 | 적용 |
|---|---|---|
| **구조화된 명세** | AI Lang | intent, context, constraints, output 등 YAML 구조로 명세의 뼈대를 잡음 |
| **완전성 + 채점** | SEDAI | 명세서 완전성을 0~100점으로 자체 채점. 95점 미만이면 부족한 부분을 명시하고 사용자에게 보완 요청 |
| **실행 계획 구조** | Harness | Work Phases, Validation Plan, Definition of Done으로 실행 가능한 수준까지 구체화 |

---

## 2. 명세서 형식

`.dev/<폴더>/plan.md`에 다음 형식으로 작성한다:

```markdown
# <기능/프로젝트 이름> 구현 명세서

상태: Draft | Reviewed | Ready | Active | Completed
작성일: YYYY-MM-DD
명세 점수: __/100

---

## Intent
- 구현 목적을 한 문장으로 (AI Lang의 intent)
- intent 유형: CREATE | UPDATE | TRANSFORM | MIGRATE | DEBUG | REFACTOR

## Context
- domain: <도메인>
- language: <언어>
- framework: <프레임워크>
- database: <DB>
- 관련 .dev/ 참조: [<이름>](./<경로>)

## Current State (현재 상태)
- 지금 구현되어 있는 것
- 현재의 문제점이나 부족한 점
- (없으면 "신규 구현" 명시)

## Expected Result (기대 결과)
- 구현 완료 후 달성해야 할 상태
- 사용자가 할 수 있게 되는 것

## Scope (범위)
### 포함
- 이 명세서가 다루는 것

### 제외 (Non-Goals)
- 이 명세서가 다루지 않는 것

## Users and Roles
- 관련 사용자 유형과 역할

## Key Flows (핵심 흐름)
### 흐름 1: <이름>
1. 단계별 사용자/시스템 행동
2. ...

### 흐름 2: <이름>
1. ...

## Entities (데이터 모델)
- 엔티티 이름, 필드, 관계
- 테이블 스키마 또는 타입 정의

## Architecture (구현 아키텍처)
### 파일 구조
- 생성/수정할 파일 목록과 역할

### 계층 구조
- 컨트롤러 → 서비스 → 리포지토리 등 의존성 방향

### 핵심 구현 코드
- 주요 함수/클래스의 시그니처와 핵심 로직
- 의사코드 또는 실제 코드 스니펫

## Constraints (제약사항)
### must (반드시)
- 반드시 지켜야 하는 규칙

### should (권장)
- 가능하면 지켜야 하는 규칙

### must_not (금지)
- 절대 하지 말아야 하는 것

## Work Phases (작업 단계)
### Phase 1: <이름>
- 작업 내용
- 산출물
- 검증 방법

### Phase 2: <이름>
- ...

## Validation Plan (검증 계획)
- 각 단계별 무엇을 어떻게 검증하는가
- 테스트 케이스 목록

## Risks (리스크)
- 예상 리스크와 대응 방안

## Acceptance Criteria (완료 기준)
- [ ] 체크리스트 형태의 완료 조건

## Decision Log (의사결정 기록)
| 일자 | 결정 | 근거 |

---

## 명세 완전성 점수: __/100
### 채점 근거:
- ...
### 부족한 부분:
- ...
```

## 3. 명세서 작성 프로세스

### 3.1 정보 수집

1. **ARGUMENTS 분석**: 사용자 요청에서 핵심 의도를 파악한다
2. **리포지토리 조사**: 현재 코드베이스를 읽어 현재 상태를 파악한다
3. **기존 .dev/ 참조**: 이전에 `/dev:copy`로 만든 참조 문서가 있으면 반드시 읽는다
4. **기술 스택 확인**: 패키지 매니저 파일, 설정 파일 등을 읽어 실제 스택을 파악한다

### 3.2 명세서 초안 작성

위 형식에 따라 `.dev/<폴더>/plan.md`를 작성한다.

**핵심 규칙:**
- 모호한 표현 금지. "적절히 처리한다" 대신 정확히 어떻게 처리하는지 명시
- 핵심 함수/클래스의 시그니처와 로직을 코드 수준으로 기술
- 파일 경로를 구체적으로 명시
- Current State와 Expected Result의 차이가 명확해야 한다

### 3.3 자체 채점 (SEDAI 원칙)

명세서 작성 후 반드시 **자체 채점**을 수행한다.

채점 기준:

| 항목 | 배점 | 기준 |
|---|---|---|
| Intent 명확성 | 10 | 무엇을 왜 만드는지 한 문장으로 설명 가능한가 |
| Context 완전성 | 10 | 기술 스택, 참조 문서, 의존성이 모두 명시되었는가 |
| Key Flows 구체성 | 15 | 사용자 흐름이 단계별로 구체적인가 |
| Entities 완전성 | 10 | 데이터 모델, 필드, 관계가 모두 정의되었는가 |
| Architecture 실행성 | 15 | 파일 구조, 계층, 핵심 코드가 포함되었는가 |
| Constraints 명확성 | 10 | must/should/must_not이 구체적인가 |
| Work Phases 실행성 | 15 | 각 단계의 산출물과 검증 방법이 있는가 |
| Acceptance Criteria | 10 | 완료 기준이 체크리스트로 검증 가능한가 |
| 전체 일관성 | 5 | 문서 내 모순이나 누락이 없는가 |

**95점 이상: Ready** — `/dev:execute`로 실행 가능
**80~94점: Reviewed** — 부족한 부분을 명시하고 사용자에게 보완 요청
**80점 미만: Draft** — 핵심 정보가 부족. 추가 정보를 반드시 요청

### 3.4 부족한 부분 처리

채점 결과 95점 미만이면:

1. 부족한 항목을 명확히 나열한다
2. 사용자에게 보완할 정보를 구체적으로 요청한다
3. 추측으로 채우지 않는다 (SEDAI 원칙: "No interpretation, no assumption")
4. 사용자가 보완 정보를 제공하면 명세서를 업데이트하고 재채점한다

예시:
```
명세 완전성 점수: 82/100

부족한 부분:
- Entities (-8): 댓글 테이블의 필드 정의가 누락됨. parent_id로 대댓글을 지원할 것인지 확인 필요
- Architecture (-5): 이미지 업로드 시 저장소(로컬/S3/Firebase Storage) 미결정
- Constraints (-5): 게시글 본문의 최대 길이, 이미지 최대 개수 미정의

다음 정보를 알려주시면 95점 이상으로 보완할 수 있습니다:
1. 대댓글 지원 여부와 최대 깊이
2. 이미지 저장소 선택
3. 본문 최대 길이와 이미지 최대 개수
```

## 4. 기존 참조 문서 활용

이전에 `/dev:copy`로 만든 참조 문서가 있으면 반드시 활용한다.

```markdown
## Context
- 관련 .dev/ 참조:
  - [기존인증](./../기존인증/README.md) — 현재 세션 기반 인증 구조
  - [게시판참조](./../게시판참조/board.md) — 참조할 게시판 패턴
```

참조 문서에서 가져오는 것:
- 핵심 로직 패턴
- 데이터 모델 구조
- 기존 코드의 제약사항
- 검증된 구현 방식

## 5. 보조 문서 생성

명세서가 복잡할 경우 보조 문서를 추가로 생성한다:

```text
.dev/<폴더>/
├── plan.md                 ← 메인 구현 명세서 (필수)
├── entities.md             ← 상세 데이터 모델 (plan.md에서 링크)
├── flows.md                ← 상세 사용자 흐름 (plan.md에서 링크)
├── api-spec.md             ← API 엔드포인트 명세 (plan.md에서 링크)
└── ui-spec.md              ← 화면 명세 (plan.md에서 링크)
```

보조 문서를 만드는 기준:
- plan.md가 1700줄을 초과할 때
- 특정 영역(데이터 모델, API 등)이 독립적으로 참조될 필요가 있을 때
- 메인 문서에서 Progressive Disclosure로 연결하는 것이 더 읽기 좋을 때

---

## 6. 실전 작업 흐름

```
1. ARGUMENTS에서 구현 대상과 저장 이름을 파악한다
2. 리포지토리를 조사한다 (현재 상태, 기술 스택, 기존 .dev/ 문서)
3. .dev/<이름>/ 디렉터리를 생성한다
4. plan.md 명세서를 작성한다
5. 자체 채점한다 (0~100점)
6. 95점 이상이면 "Ready"로 표시하고 사용자에게 보고한다
7. 95점 미만이면 부족한 부분을 명시하고 사용자에게 보완을 요청한다
8. 보완 후 재채점한다
```

---

## 7. 완료 기준

- [ ] `.dev/<이름>/plan.md`가 존재하는가
- [ ] Intent가 한 문장으로 명확한가
- [ ] Key Flows가 단계별로 구체적인가
- [ ] Entities가 필드 수준으로 정의되었는가
- [ ] Architecture에 파일 구조와 핵심 코드가 포함되었는가
- [ ] Constraints가 must/should/must_not으로 구분되었는가
- [ ] Work Phases에 각 단계의 산출물과 검증 방법이 있는가
- [ ] 자체 채점 점수가 95점 이상인가
- [ ] 이 명세서만 읽고 `/dev:execute`로 바로 구현을 시작할 수 있는가

**명세서만 읽고 구현을 시작할 수 없다면, 계획이 완료된 것이 아니다.**
