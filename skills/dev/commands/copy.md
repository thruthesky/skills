# /dev:copy — 참조 문서 복사·생성

## 0. 필수 입력: ARGUMENTS

**이 명령을 사용하려면 반드시 ARGUMENTS를 제공해야 한다.**

ARGUMENTS가 없으면 사용자에게 다음을 요청하라:

```
/dev:copy를 사용하려면 다음 정보를 프롬프트에 포함해 주세요:

필수:
  - 복사 대상: 무엇을 복사할 것인가 (파일 경로, 프로젝트 경로, 개념, URL 등)

선택:
  - 저장 이름: .dev/<이름>/ 폴더에 저장할 이름
    (생략하면 복사 대상에서 자동 추론하여 이름을 결정)
  - 복사 범위: 전체 / 특정 기능만 / 핵심 로직만
  - 초점 영역: 어떤 부분에 초점을 맞출 것인가
```

### 저장 이름 결정 규칙

사용자가 이름을 지정하면 그 이름을 사용한다.
사용자가 이름을 생략하면 복사 대상에서 자동 추론한다:
- 파일/디렉터리 경로 → 마지막 디렉터리명 또는 파일명 (예: `src/auth/` → `auth`)
- URL → 리포지토리명 또는 프로젝트명 (예: `github.com/xxx/board` → `board`)
- 개념/설명 → 핵심 키워드를 kebab-case로 (예: "로그인 기능" → `login`)

ARGUMENTS 예시:
```
예시 1: "/dev:copy src/auth/ 인증기능 — 로그인, 회원가입 로직을 참조 문서로 만들어줘"
예시 2: "/dev:copy src/auth/ — 이름 생략 → .dev/auth/ 에 저장"
예시 3: "/dev:copy https://github.com/xxx/yyy board — 게시판 프로젝트 구조와 핵심 로직을 복사해줘"
예시 4: "/dev:copy lib/widgets/chat.dart — 이름 생략 → .dev/chat/ 에 저장"
```

---

## 1. 이 명령의 목적

`/dev:copy`는 기존 코드, 프로젝트, 개념, 로직을 **분석**하여 `.dev/<폴더>/`에 **에이전트가 참조할 수 있는 구조화된 문서**로 만든다.

핵심 원칙 (3가지 개념 융합):

| 원칙 | 출처 | 적용 |
|---|---|---|
| **소스코드 포함 필수** | SEDAI | 참조 문서에 핵심 코드를 반드시 포함. 문서만 보고 전체 기능을 복구할 수 있어야 함 |
| **구조화된 형식** | AI Lang | intent, context, entities 등 구조화된 키워드로 정리 |
| **Progressive Disclosure** | Harness | 메인 문서 → 서브 문서로 점진적 탐색 가능하게 구성 |

## 2. 작업 프로세스

### 2.1 대상 분석

먼저 복사 대상을 철저히 분석한다:

- 파일/디렉터리 구조 파악
- 진입점(entry point) 식별
- 핵심 로직 흐름 추적
- 데이터 모델/엔티티 파악
- 외부 의존성 확인
- 설정/환경 파일 확인

### 2.2 문서 생성 구조

`.dev/<폴더>/` 안에 다음 구조로 문서를 생성한다:

```text
.dev/<폴더>/
├── README.md              ← 메인 문서 (전체 요약 + 서브 문서 링크)
├── architecture.md        ← 구조와 흐름 (필요 시)
├── <기능1>.md             ← 기능별 핵심 로직 + 소스코드
├── <기능2>.md             ← 기능별 핵심 로직 + 소스코드
└── entities.md            ← 데이터 모델/엔티티 (필요 시)
```

규모에 따라 조절:
- **작은 복사** (단일 파일/함수): `README.md` 하나로 충분
- **중간 복사** (한 기능): `README.md` + 기능 문서 1~2개
- **큰 복사** (프로젝트 전체): 위 전체 구조

### 2.3 각 문서의 필수 포함 항목

모든 참조 문서는 **SEDAI 원칙**에 따라 다음을 반드시 포함한다:

```markdown
# <기능/모듈 이름>

## Intent
- 이 기능이 해결하는 문제 (AI Lang의 intent 개념)

## Context
- 기술 스택, 프레임워크, 언어
- 의존하는 외부 서비스/라이브러리

## 핵심 로직
- 비즈니스 로직의 정확한 동작 방식
- 데이터 흐름과 처리 순서
- 조건 분기와 엣지 케이스

## 핵심 소스코드
- 실제 구현된 코드 (전체 또는 핵심 부분)
- 파일 경로 명시
- 코드 내 한글 주석으로 로직 설명

## Entities
- 관련 데이터 모델/타입/스키마

## Constraints
- 반드시 지켜야 하는 제약사항
- must / should / must_not 구분

## 파일 목록
- 관련된 모든 소스 파일 경로
```

### 2.4 메인 문서 (README.md) 작성 규칙

메인 문서는 **Harness의 AGENTS.md 원칙**을 따른다:

- 전체 요약을 간결하게 (지도 역할)
- 각 서브 문서로의 링크와 **100~120단어 요약** 포함
- 기술 스택, 진입점, 핵심 제약을 명시

```markdown
# <프로젝트/기능 이름> 참조 문서

## 개요
- 무엇을 하는 코드인가 (1~2문장)
- 기술 스택: ...
- 원본 위치: ...

## 문서 구조
### 인증 기능 → [auth.md](./auth.md)
Firebase Authentication 기반 사용자 인증 처리. 로그인, 회원가입,
토큰 검증, 세션 관리의 전체 흐름을 다룬다. 프론트엔드의
firebase_user_ready() 콜백과 백엔드의 assert_user() 함수를
중심으로 인증 상태 관리 패턴을 설명한다. (100~120단어)

### 게시판 기능 → [board.md](./board.md)
게시글 CRUD와 댓글 시스템의 전체 구현. ... (100~120단어)

## 핵심 제약
- ...
```

### 2.5 문서 크기 제한

**각 문서는 최대 1700줄로 제한한다.**

1700줄을 초과하면:
1. 논리적 서브 카테고리로 분리
2. 파일명 규칙: `<메인>-<서브>.md` (예: `auth-login.md`, `auth-register.md`)
3. 메인 문서에 각 서브 문서의 100~120단어 요약 포함

---

## 3. 실전 작업 흐름

```
1. ARGUMENTS에서 복사 대상과 저장 이름을 파악한다
2. 대상 코드/프로젝트를 읽고 분석한다
3. .dev/<이름>/ 디렉터리를 생성한다
4. 규모에 맞는 문서 구조를 결정한다
5. 각 문서에 핵심 로직 + 핵심 소스코드를 반드시 포함하여 작성한다
6. README.md에 전체 요약과 서브 문서 링크를 작성한다
7. 사용자에게 생성된 문서 목록을 보고한다
```

---

## 4. 완료 기준

- [ ] `.dev/<이름>/README.md`가 존재하고 전체 지도 역할을 하는가
- [ ] 모든 핵심 로직이 문서에 포함되어 있는가
- [ ] 모든 핵심 소스코드가 문서에 포함되어 있는가 (SEDAI 원칙)
- [ ] 파일 경로가 명시되어 있는가
- [ ] 이 문서만 보고 전체 기능을 복구/재생성할 수 있는가
- [ ] 에이전트가 Progressive Disclosure로 탐색할 수 있는 구조인가
- [ ] 각 문서가 1700줄 이하인가

**이 문서만 보고 전체 기능을 완벽하게 복구할 수 없다면, 복사가 완료된 것이 아니다.**
