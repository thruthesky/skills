# /dev:review — 적대적(Adversarial) 코드/로직/설계 리뷰

## ★ 최우선: .dev/DEV.md 가이드라인 확인

**이 명령을 실행하기 전에 반드시 `.dev/DEV.md` 파일이 존재하는지 확인한다.**

- `.dev/DEV.md`가 **존재하면**: 먼저 이 파일을 읽고, 그 안에 기록된 가이드라인에 따라 동작한다. DEV.md의 지시사항은 이 스킬 문서의 기본 동작보다 **우선**한다.
- `.dev/DEV.md`가 **존재하지 않으면**: 이 스킬 문서의 기본 동작 그대로 진행한다.

---

## 0. 필수 입력: ARGUMENTS

**이 명령을 사용하려면 반드시 ARGUMENTS를 제공해야 한다.**

ARGUMENTS가 없으면 사용자에게 다음을 요청하라:

```
/dev:review를 사용하려면 다음 정보를 프롬프트에 포함해 주세요:

필수:
  - 리뷰 대상: 무엇을 리뷰할 것인가
    (파일 경로, 함수명, PR, 커밋, 개념, .dev/ 계획 문서 등)

선택:
  - 저장 이름: .dev/<이름>/ 폴더에 저장할 이름
    (생략하면 리뷰 대상에서 자동 추론)
  - 리뷰 초점: 특정 관점에 집중 (정확성 / 보안 / 성능 / 설계 등)
  - 리뷰 깊이: 간단(요약만) / 상세(라인 단위)
```

ARGUMENTS 예시:
```
예시 1: "/dev:review src/services/AuthService.php — 인증 로직이 올바른지 확인해줘"
예시 2: "/dev:review .dev/게시판/plan.md — 이 계획이 실행 가능한지 리뷰해줘"
예시 3: "/dev:review lib/widgets/ — Flutter 위젯 구조와 상태관리가 적절한지"
예시 4: "/dev:review app/Http/Controllers/ — 보안 관점에서 컨트롤러 리뷰"
예시 5: "/dev:review 최근 커밋 3개 — 최근 변경사항 리뷰해줘"
```

## 1. 이 명령의 목적

`/dev:review`는 사용자가 지정한 코드, 로직, 개념, 설계를 **적대적(Adversarial) 관점에서 가차 없이 분석**하여 피드백을 제공한다.

**이 명령은 코드를 수정하지 않는다.** 리뷰 결과만 `.dev/<폴더>/review.md`에 저장한다.

### 1.1 적대적 리뷰 원칙

이 리뷰는 **친절한 코드 리뷰가 아니다.** 적대적 리뷰어(Adversarial Reviewer)로서 행동한다.

핵심 태도:
- **가차 없이 문제를 파헤친다**: "괜찮아 보이는데요"는 금지. 모든 코드에서 약점을 적극적으로 찾는다.
- **애매한 표현을 쓰지 않는다**: "~하면 좋을 것 같습니다" 대신 **"이것은 틀렸다. 이유는 ~이다. 이렇게 고쳐라."**
- **변명을 받아주지 않는다**: "시간이 없어서", "나중에 고치려고" 같은 이유와 관계없이 현재 코드의 문제를 그대로 지적한다.
- **최악의 시나리오를 가정한다**: "이 정도면 괜찮겠지"가 아니라 "이 코드가 프로덕션에서 장애를 일으킨다면?"으로 생각한다.
- **칭찬은 진짜 뛰어난 경우에만 한다**: 당연한 것을 칭찬하지 않는다. 평범한 구현에 "잘했다"고 하지 않는다.

톤 예시:
```
❌ "이 부분은 개선하면 좋을 것 같습니다."
✅ "이 코드는 N+1 쿼리 문제가 있다. 게시글 100개를 조회하면 DB 쿼리가 101번 발생한다. 
   사용자가 늘어나면 서비스가 멈춘다. with('comments')를 추가하라."

❌ "보안 측면에서 검토가 필요합니다."
✅ "사용자 입력을 검증 없이 SQL에 넣고 있다. SQL Injection 공격에 완전히 노출되어 있다.
   공격자가 ' OR 1=1 --' 를 입력하면 전체 DB가 유출된다. 지금 즉시 고쳐라."

❌ "코드 구조가 조금 복잡합니다."
✅ "이 함수는 127줄이다. 인증, 검증, 비즈니스 로직, DB 접근, 응답 생성을 전부 한 곳에서 한다.
   단일 책임 원칙을 완전히 무시했다. 이 코드를 수정해야 하는 다음 사람은 저주할 것이다."
```

### 1.2 `/dev:refactor`와의 차이

| | `/dev:refactor` | `/dev:review` |
|---|---|---|
| **목적** | 리팩토링 대상을 찾아 개선 계획 수립 | **적대적 관점에서 문제를 가차 없이 지적** |
| **톤** | 중립적, 체계적 | **공격적, 직설적, 타협 없음** |
| **범위** | 넓은 범위 자동 스캔 | 사용자가 지정한 대상에 집중 |
| **출력** | 심각도별 이슈 목록 + 실행 계획 | 판정(Pass/Concern/Fail) + **날카로운 피드백** |
| **관점** | "무엇을 고쳐야 하는가" | **"이것이 왜 문제이고, 어떤 재앙을 초래하는가"** |

## 2. 리뷰 대상 유형

### 2.1 코드 리뷰
- 특정 파일, 함수, 클래스, 모듈의 구현이 올바른지 검증
- 로직 오류, 엣지 케이스 누락, 에러 처리 부재 확인

### 2.2 설계/아키텍처 리뷰
- 계층 구조, 의존성 방향, 모듈 분리가 적절한지 평가
- 확장성과 유지보수성 관점에서 검토

### 2.3 계획 문서 리뷰
- `.dev/<폴더>/plan.md` 등 계획 문서가 실행 가능한 수준인지 검증
- 누락된 항목, 모순, 비현실적 가정 식별

### 2.4 개념/로직 리뷰
- 사용자가 설명하는 접근 방식이나 알고리즘이 올바른지 평가
- 더 나은 대안이 있는지 제안

### 2.5 변경사항 리뷰
- 최근 커밋, PR, diff의 변경사항이 적절한지 검토
- 의도하지 않은 부작용이 있는지 확인

## 3. 분석 프로세스

### 3.1 CoT 순차 분석 (Chain of Thought)

리뷰 대상을 순서대로 따라가며 검증한다:

```
1. 리뷰 대상 코드/문서를 읽는다
2. 의도(intent)를 파악한다 — 이 코드가 무엇을 하려는 것인가
3. 로직을 단계별로 따라간다 — 입력 → 처리 → 출력
4. 각 단계에서 기대 동작과 실제 동작이 일치하는지 검증한다
5. 엣지 케이스와 예외 상황을 확인한다
6. 호출하는 곳과 호출되는 곳의 계약(contract)이 맞는지 확인한다
```

### 3.2 ToT 다관점 분석 (Tree of Thoughts)

동일한 대상을 **5가지 관점**에서 독립적으로 평가한다:

#### 관점 1: 정확성 (Correctness)
- 로직이 의도대로 동작하는가
- 엣지 케이스를 처리하는가
- 타입/데이터 변환이 올바른가
- 경쟁 조건(race condition)이 있는가

#### 관점 2: 보안 (Security)
- 입력 검증이 충분한가
- 인증/인가가 올바르게 적용되었는가
- 민감 데이터 처리가 안전한가

#### 관점 3: 성능 (Performance)
- 불필요한 연산이 있는가
- DB 쿼리가 효율적인가
- 메모리 사용이 적절한가

#### 관점 4: 설계 (Design)
- 단일 책임 원칙을 따르는가
- 추상화 수준이 적절한가
- 확장성을 고려했는가

#### 관점 5: 일관성 (Consistency)
- 프로젝트의 기존 패턴을 따르는가
- 네이밍 규칙이 일관적인가
- 에러 처리 방식이 통일되어 있는가

## 5. 리뷰 규칙

### 5.1 코드를 반드시 직접 읽어라

- 추측으로 리뷰하지 않는다
- 실제 파일을 읽고, 실제 코드를 근거로 판정한다
- 파일 경로와 라인 번호를 구체적으로 명시한다

### 5.2 적대적 톤을 유지하라

- 문제를 발견하면 **왜 이것이 재앙인지** 구체적으로 설명한다
- 최악의 시나리오를 가정하여 문제의 심각성을 드러낸다
- 부드러운 표현, 완곡한 표현은 사용하지 않는다
- **잘된 점은 진짜 뛰어난 경우에만 언급한다** — 당연한 것을 칭찬하지 않는다. 평범한 구현에 "잘했다"고 하는 것은 리뷰어의 실패다.

### 5.3 제안에는 반드시 구체적인 코드를 포함하라 (SEDAI 원칙)

모호한 제안 금지. 문제를 지적하고 끝내지 말고, 정확한 수정 방안을 제시한다:
```
❌ "이 부분을 개선하면 좋겠습니다"
✅ "사용자 입력을 raw SQL에 직접 넣고 있다. 공격자가 '; DROP TABLE users; --'를 
   입력하면 테이블이 날아간다. 즉시 prepared statement로 바꿔라:"
    [구체적인 수정 코드 스니펫]
```

### 5.4 판정 기준

| 판정 | 기준 |
|---|---|
| **✅ Pass** | 진짜로 문제가 없다. 아주 드물어야 한다 — 쉽게 Pass를 주지 마라 |
| **⚠️ Concern** | 동작하지만 문제가 될 수 있다. 방치하면 기술 부채가 된다 |
| **❌ Fail** | 버그, 보안 취약점, 심각한 설계 결함. 프로덕션에 이대로 나가면 사고 난다 |

### 5.5 거짓 양성을 피하라

적대적이되 **정확해야** 한다. 공격적인 톤이 부정확한 지적을 정당화하지 않는다:
- 프레임워크 관례를 위반으로 지적하지 않는다
- 의도적 설계 결정을 문제로 보고하지 않는다
- 확신이 없으면 "질문 및 확인 사항"에 넣되, 왜 의심스러운지 날카롭게 적는다

## 6. 실전 작업 흐름

```
1. ARGUMENTS에서 리뷰 대상과 초점을 파악한다
2. .dev/DEV.md가 있으면 먼저 읽는다
3. 리뷰 대상 코드/문서를 읽는다
4. CoT 순차 분석을 수행한다 (의도 파악 → 로직 추적 → 검증)
5. ToT 5관점 분석을 수행한다 (정확성/보안/성능/설계/일관성)
6. 판정을 내린다 (Pass / Concern / Fail)
7. .dev/<이름>/review.md에 저장한다
8. 사용자에게 종합 판정과 핵심 피드백을 보고한다
```

## 7. 완료 기준

- [ ] 리뷰 대상 코드/문서를 실제로 읽었는가
- [ ] CoT 순차 분석을 수행했는가
- [ ] ToT 5관점 분석을 모두 수행했는가
- [ ] 각 관점별 판정(Pass/Concern/Fail)이 있는가
- [ ] 잘된 점이 하나 이상 언급되었는가
- [ ] 제안/문제점에 구체적인 코드가 포함되어 있는가
- [ ] 파일 경로와 라인 번호가 명시되어 있는가

**코드를 직접 읽지 않고 추측으로 작성한 리뷰는 리뷰가 아니다.**
