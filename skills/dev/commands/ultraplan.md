# Ultraplan — 초정밀 사전 계획 커맨드

## 목적

현재 프로젝트에서 코딩/기능추가/디자인/개발을 시작하기 **전에**, 아주 상세하고 완전한 구현 계획을 수립하는 사전 작업 커맨드이다.
최종 산출물은 **`/dev:plan`과 동일한 형식**(`plan.md`)으로 작성하되, 일반 `/dev:plan`보다 **한 층 더 업그레이드된 심층 계획**을 세운다.

**핵심 차별점**: 일반 `/dev:plan`은 단일 에이전트가 순차적으로 계획을 세우지만, Ultraplan은 **에이전트 팀(또는 서브 에이전트 병렬)을 활용한 3단계 심층 분석 → 비주얼 다이어그램 → dev:plan 명세서** 순서로 진행하여, 더 강력하고 정교한 계획을 생성한다.

**Ultraplan vs dev:plan 비교:**

| 항목 | `/dev:plan` | `/ultraplan` |
|------|------------|-------------|
| 분석 방식 | 단일 에이전트 순차 분석 | **에이전트 팀/서브 에이전트 병렬 분석** |
| 분석 깊이 | 표면 수준 정보 수집 | **3개 관점 독립 탐색 + 비평 에이전트 통합** |
| 비주얼 산출물 | 없음 | **ASCII 다이어그램 필수 (아키텍처 + 흐름도)** |
| 리스크 분석 | Risks 섹션에 간략 기술 | **Tree-of-Thoughts 방식 심층 리스크 분석** |
| 최종 산출물 | `.dev/<이름>/plan.md` | `.dev/<이름>/plan.md` (동일 형식, 더 높은 품질) |
| 명세 점수 목표 | 95점 이상 | **95점 이상 필수 (미달 시 보완 후 재작성)** |

## Phase 1: 프로젝트 컨텍스트 수집 (필수)

### 1.1 필수 참조 문서 읽기

다음 문서들을 **반드시** 읽고 프로젝트 현재 상태를 완전히 파악한다:

| 순서 | 대상 | 파악할 내용 |
|------|------|------------|
| 1 | `CLAUDE.md` | 기술 스택, 아키텍처 원칙, 코딩 규칙, 필수 워크플로우, 타입 검사 규칙 |
| 2 | **현재 프로젝트의 스킬** (`SKILL.md` + `references/*.md`) | 프로젝트 전체 구조, 참조 문서 인덱스, 핵심 패턴. CLAUDE.md에 프로젝트 스킬 경로가 명시되어 있으면 해당 스킬을 읽고, 없으면 `.claude/skills/` 하위를 탐색하여 관련 스킬을 찾는다 |
| 3 | `.dev/` 디렉토리 | 기존 참조/계획 문서 중 ARGUMENTS와 관련된 것 |
| 4 | 관련 소스코드 | Controller, Service, Repository, Entity, 페이지 등 실제 코드 |

**프로젝트 스킬 탐색 규칙:**
1. CLAUDE.md를 먼저 읽어 프로젝트 설명과 스킬 경로가 있는지 확인한다
2. 프로젝트 스킬이 있으면 `SKILL.md`와 관련 `references/*.md`를 읽어 프로젝트 전체 구조를 파악한다
3. 프로젝트 스킬이 없으면 CLAUDE.md의 정보와 소스코드 직접 탐색으로 컨텍스트를 수집한다

### 1.2 프로젝트 현재 상태 종합

수집한 정보를 다음 체크리스트로 정리한다:

```
[ ] 기술 스택 확인 (언어, 프레임워크, DB, 프론트엔드 등 — CLAUDE.md에서 파악)
[ ] 아키텍처 레이어 파악 (MVC, Clean Architecture, 모듈 구조 등)
[ ] 프로젝트 고유 패턴 이해 (인증, 멀티테넌트, 라우팅 등 — 프로젝트 스킬에서 파악)
[ ] ARGUMENTS 관련 기존 코드/문서 파악 완료
[ ] 영향받는 파일 목록 식별
```

## Phase 2: 에이전트 팀 Deep Analysis

ARGUMENTS를 **3개의 독립적인 관점**에서 **에이전트 팀을 활용하여 병렬로** 탐색하고, **1개의 비평 에이전트**가 결과를 통합·평가한다.

### 에이전트 실행 전략

```
┌─────────────────────────────────────────────────────────────────┐
│  에이전트 팀 실행 전략 (우선순위 순서)                             │
│                                                                 │
│  1순위: 에이전트 팀(Agent Team)이 사용 가능하면                   │
│         → 에이전트 팀으로 3개 에이전트를 동시 실행                │
│                                                                 │
│  2순위: 에이전트 팀이 불가능하면                                  │
│         → 서브 에이전트(Agent 도구)를 3개 동시 호출하여           │
│           병렬로 분석 수행                                       │
│                                                                 │
│  ※ 어떤 경우든 3개 에이전트는 반드시 병렬로 실행해야 한다.       │
│    순차 실행은 금지된다.                                         │
└─────────────────────────────────────────────────────────────────┘
```

### 2.1 병렬 탐색 에이전트 (3개 동시 실행)

```
┌─────────────────────────────────────────────────────────────────────┐
│                        ARGUMENTS 분석                               │
├─────────────────┬─────────────────────┬─────────────────────────────┤
│                 │                     │                             │
│  Agent A        │  Agent B            │  Agent C                   │
│  [아키텍처]     │  [데이터/흐름]      │  [리스크/엣지케이스]       │
│                 │                     │                             │
│  ● 아키텍처에서 │  ● 데이터 모델      │  ● 보안 취약점             │
│    어디에 위치?  │    (Entity/필드)     │  ● 성능 병목               │
│  ● 기존 코드와  │  ● 사용자 흐름      │  ● 프로젝트 고유 제약 위반 │
│    의존성 관계   │    (step-by-step)   │  ● 타입 안전성 위험        │
│  ● 파일 생성/   │  ● API 엔드포인트   │  ● 동시성/경합 조건        │
│    수정 목록     │  ● DB 마이그레이션  │  ● 기존 기능 회귀          │
│  ● 라우팅 변경  │  ● 검색/캐시 영향   │  ● 잘못된 가정 목록        │
│                 │                     │                             │
└─────────────────┴─────────────────────┴─────────────────────────────┘
```

**각 에이전트는 Chain-of-Thought(CoT) 방식으로 단계별 추론한다.**

#### Agent A: 아키텍처 분석 (CoT)

```
사고 체인:
1. ARGUMENTS가 요구하는 기능은 무엇인가?
2. 현재 아키텍처 구조에서 어디에 위치하는가?
   → 각 레이어(Controller/Service/Repository/Entity 등)에 변경이 필요한가?
3. 기존 코드 중 재사용 가능한 것은?
4. 새로 생성해야 할 파일은?
5. 기존 파일 중 수정이 필요한 것은?
6. 라우팅에 변경이 필요한가?
7. 프론트엔드(페이지/컴포넌트/스타일) 변경은?
```

#### Agent B: 데이터/흐름 분석 (CoT)

```
사고 체인:
1. 어떤 데이터가 필요한가? (Entity, 테이블, 필드)
2. DB 마이그레이션이 필요한가?
3. 사용자 흐름은 어떻게 되는가? (step 1 → step 2 → ...)
4. API 엔드포인트가 필요한가? (메서드, URL, 파라미터, 응답)
5. 검색/캐시/인덱스에 영향이 있는가?
6. 알림(푸시, 이메일 등)이 필요한가?
7. 다국어(i18n) 문자열이 필요한가?
```

#### Agent C: 리스크/엣지케이스 분석 (Tree-of-Thoughts)

```
사고 트리:
├── 보안
│   ├── XSS 가능성은?
│   ├── SQL Injection 가능성은?
│   ├── CSRF 대응은?
│   └── 인증/권한 우회 가능성은?
├── 프로젝트 고유 제약
│   ├── CLAUDE.md에 명시된 규칙을 모두 준수하는가?
│   ├── 프로젝트 스킬에 정의된 패턴과 일치하는가?
│   └── 기존 아키텍처 원칙에 위배되지 않는가?
├── 타입 안전성
│   ├── nullable 반환값 목록은?
│   ├── union 타입 처리가 필요한 곳은?
│   └── 타입 에러 발생 가능 지점은?
├── 성능
│   ├── N+1 쿼리 가능성은?
│   ├── 대량 데이터 처리 시 문제는?
│   └── 인덱스가 필요한 쿼리는?
└── 회귀
    ├── 기존 테스트에 영향은?
    ├── 기존 API 호환성은?
    └── 기존 UI/UX에 변경은?
```

### 2.2 비평 에이전트 (통합·평가)

3개 에이전트의 분석 결과를 받아 다음을 수행한다:

```
┌────────────────────────────────────────────────────────┐
│  비평 에이전트 — 통합 평가                              │
│                                                        │
│  1. 3개 분석 결과 간 모순 검출                          │
│     → 모순이 있으면 근거를 비교하여 올바른 쪽 채택      │
│                                                        │
│  2. 누락 항목 식별                                      │
│     → 3개 에이전트 모두 놓친 관점이 있는지 확인         │
│                                                        │
│  3. 우선순위 정리                                       │
│     → Critical / High / Medium / Low로 분류             │
│                                                        │
│  4. 실행 가능성 판단                                    │
│     → 현재 프로젝트 상태에서 구현 가능한지 평가         │
│     → 선행 작업(마이그레이션 등)이 필요한지 식별        │
│                                                        │
│  5. 통합 분석 보고서 생성                               │
│     → Phase 3의 입력 데이터로 사용                      │
└────────────────────────────────────────────────────────┘
```

## Phase 3: 비주얼 계획 생성 + plan.md 명세서 작성

### 3.1 비주얼 다이어그램 생성 (필수 — Ultraplan의 핵심 차별점)

Phase 2의 분석 결과를 바탕으로 **최소 2개 이상**의 다이어그램을 생성한다.

**비주얼 다이어그램은 Ultraplan의 핵심 특징이다.** 복잡한 구현 계획을 텍스트로만 설명하면 이해하기 어렵다. 다이어그램을 통해 사용자가 계획의 전체 구조와 흐름을 **한눈에** 파악할 수 있도록 한다.

#### 필수 다이어그램 유형

| 다이어그램 | 형식 | 용도 |
|-----------|------|------|
| **아키텍처 다이어그램** | ASCII | 계층 구조, 파일 구조, 의존성 관계 시각화 |
| **흐름도 (Flow Chart)** | ASCII | 사용자 흐름 또는 데이터 흐름 step-by-step 시각화 |

#### 선택 다이어그램 유형 (필요 시 추가)

| 다이어그램 | 형식 | 용도 |
|-----------|------|------|
| 시퀀스 다이어그램 | ASCII | 클라이언트↔서버↔DB 간 요청/응답 순서 |
| 데이터 모델 (ERD) | ASCII | Entity 간 관계, 필드, FK |
| 상태 다이어그램 | ASCII | 상태 전이 (예: 주문 상태, 신고 처리 상태) |
| 컴포넌트 다이어그램 | ASCII | 프론트엔드 컴포넌트 구조 |

#### ASCII 다이어그램 예시

**아키텍처 다이어그램 예시:**
```
                    ┌──────────────┐
                    │   Browser    │
                    └──────┬───────┘
                           │ HTTP Request
                           ▼
                    ┌──────────────┐
                    │   Router     │  (라우팅)
                    └──────┬───────┘
                           │
              ┌────────────┼────────────┐
              ▼            ▼            ▼
       ┌────────────┐ ┌────────┐ ┌──────────┐
       │ Controller │ │ Pages  │ │ Partials │
       │ (API)      │ │ (View) │ │ (Layout) │
       └─────┬──────┘ └────────┘ └──────────┘
             │
             ▼
       ┌────────────┐
       │  Service   │ ← 비즈니스 로직
       └─────┬──────┘
             │
             ▼
       ┌────────────┐
       │ Repository │ ← DB 접근
       └─────┬──────┘
             │
             ▼
       ┌────────────┐
       │  Entity    │ ← 데이터 모델
       └────────────┘
```

**흐름도 예시:**
```
  [사용자 액션]
       │
       ▼
  ┌─ 인증 확인 ─┐
  │              │
  ▼ (인증됨)    ▼ (미인증)
  권한 확인      401 반환
  │              
  ▼ (권한 있음)  
  입력값 검증    
  │              
  ├─ 유효 ──▶ Service 호출 ──▶ Repository ──▶ DB
  │                                           │
  │                                           ▼
  │                                      성공 응답
  │
  └─ 무효 ──▶ 422 반환
```

### 3.2 plan.md 명세서 작성 (핵심 — 반드시 실행)

Phase 2의 통합 분석 보고서 + Phase 3.1의 다이어그램을 기반으로,
**`/dev:plan`과 동일한 형식으로 `.dev/<이름>/plan.md` 명세서를 작성한다.**

Ultraplan은 내부적으로 `/dev:plan`과 동일한 명세서 구조를 사용하지만, 다음 항목이 추가·강화된다:

```
┌─────────────────────────────────────────────────────────────────┐
│  Ultraplan 강화 항목 (일반 dev:plan + α)                        │
│                                                                 │

```

## 주의사항

1. **절대 코드를 작성하지 않는다** — Ultraplan은 계획 수립 전용이다. 코드 구현은 `/dev:execute`로 별도 실행한다.
2. **추측으로 채우지 않는다** — 정보가 부족하면 사용자에게 구체적으로 질문한다 (SEDAI 원칙).
3. **CLAUDE.md 규칙을 반드시 준수한다** — CLAUDE.md에 명시된 타입 검사, 아키텍처 원칙, 코딩 규칙 등이 계획에 반영되어야 한다.
4. **프로젝트 스킬이 있으면 반드시 참조한다** — 프로젝트 스킬의 패턴, 로직, 구조를 계획에 일관되게 반영한다.
5. **다이어그램은 ASCII로 작성한다** — 터미널/마크다운 환경에서 바로 읽을 수 있어야 한다.
6. **최종 산출물은 `/dev:plan`과 동일한 형식으로 생성한다** — Ultraplan의 심층 분석 결과를 dev:plan 명세서 형식에 맞춰 통합한다.
7. **에이전트 병렬 실행은 필수이다** — 에이전트 팀이 가능하면 에이전트 팀을, 불가능하면 서브 에이전트를 병렬로 사용하여 3개 관점을 동시에 분석한다.
8. **비주얼 다이어그램은 필수이다** — 사용자가 계획의 전체 구조와 흐름을 한눈에 파악할 수 있도록 아키텍처 다이어그램과 흐름도를 반드시 포함한다.
